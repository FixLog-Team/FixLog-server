# Git 브랜치 규칙

## 1. 목적

- 스프린트 단위로 안정적으로 개발·테스트·배포를 진행하기 위해 브랜치 역할을 명확히 구분한다.
- **배포 기준(main)** 과 **개발 통합 기준(dev)**, **이슈 단위 개발 기준(이슈 브랜치)**, **개인 작업 기준(개인 브랜치)** 를 분리한다.

---

## 2. 브랜치 종류 및 역할

### 2.1 `main`

- **배포 기준 브랜치**
- 스프린트가 **마무리된 후** 검증 완료된 변경사항만 머지한다.
- 이 브랜치를 기준으로 배포한다.

**규칙**

- 스프린트 종료(릴리즈) 시점에만 머지를 원칙으로 한다.
- 긴급 수정(Hotfix)이 필요한 경우에는 예외 프로세스를 별도로 정의하거나, 최소한의 변경만 반영한다.

---

### 2.2 `dev`

- **개발 통합(Integration) 브랜치**
- 스프린트 도중, 개발 및 테스트가 완료된 이슈를 순차적으로 머지한다.
- 팀(또는 프로젝트)의 “현재 개발 최신 상태”를 대표한다.

**규칙**

- 개별 이슈 브랜치에서 기능이 완료되고 테스트를 통과하면 `dev`로 머지한다.
- `dev`는 항상 빌드/기본 테스트가 가능한 상태를 지향한다.

---

### 2.3 `<kan-이슈번호>-간단한-설명` (이슈 브랜치)

- **개별 이슈 단위 브랜치**
- 특정 JIRA 티켓(이슈 번호)에 대응하는 구현을 모으는 브랜치.
- 개인 작업 브랜치에서 올라온 작업들을 이 브랜치로 머지한 뒤, 통합 테스트/검증을 수행한다.

**규칙**

- 이슈 완료 시점에 `dev`로 머지한다.
- 이슈 브랜치에서는 “이슈 범위 내 변경만” 포함한다(스코프 관리).

**명명 예시**

- `kan-1234`
- (권장) 제목을 포함하려면: `kan-1234-login-timeout` 등

---

### 2.4 `<개인이니셜>/<브랜치명>` (개인 작업 브랜치)

- **개인 작업용 브랜치**
- 개인이 자유롭게 작업하고 수시로 푸시할 수 있는 브랜치.
- 검증이 끝난 작업은 이슈 브랜치로 올려 통합한다.

**규칙**

- 개인 브랜치는 작업 편의용이므로, 직접 `dev`/`main`으로 머지하지 않는다(원칙).
- 브랜치명은 팀에서 식별 가능하도록 간단하고 명확하게 작성한다.

**명명 예시**

- `hs/1234` (예: 이슈번호 기반)
- `hs/jira-1234` (예: 이슈 브랜치와 매칭)
- `hs/feature-login-timeout`

---

## 3. 권장 작업 흐름(Flow)

1. **이슈 생성(JIRA 티켓 확정)**
   - 이슈 브랜치 생성: `<kan-이슈번호>-이름`
2. **개인 작업 브랜치에서 개발**
   - 개인 브랜치 생성: `<개인이니셜>/<이슈번호 또는 작업명>`
   - 개발/커밋/푸시 반복
3. **개인 브랜치 → 이슈 브랜치로 통합**
   - 개인 작업 단위 PR/머지
   - 이슈 브랜치에서 기능 단위 테스트/검증 수행
4. **이슈 브랜치 → `dev` 머지**
   - 개발 및 테스트 완료 후 `dev`에 통합
5. **스프린트 종료 시 `dev` → `main` 머지**
   - 스프린트 릴리즈 범위 확정 및 회귀 테스트 후 `main`에 반영
   - `main` 기준 배포
   - 태그 따두기
     - 000-sprint-날짜-날짜.0
       - N번째 스프린트-날짜-날짜.핫픽스 번호

---

## 4. 브랜치 보호/머지 규칙(권장)

> 아래 항목은 프로젝트 성격에 따라 선택 적용한다.

### 4.1 머지 방식

- 기본 원칙: **리베이스 후 머지(Rebase 후 Merge)** 를 사용해 **Git 히스토리(라인)를 깔끔하게 유지**한다.
  - 개인/이슈 브랜치에서 작업한 커밋을 `dev`(또는 대상 브랜치) 최신 기준으로 **rebase**한 뒤 머지한다.
  - 가능하면 불필요한 merge commit을 줄이고, 변경 이력을 선형(linear)으로 관리한다.

**권장 운영 방식**

- PR 머지 시: 플랫폼 기능이 있다면 **Rebase and merge**(또는 이에 준하는 선형 머지) 옵션을 우선 사용
  - 우리는 github에 rebase and merge 기능 사용
- 로컬에서 rebase 수행 시: 충돌 해결 후 테스트까지 완료한 다음 머지

> 주의: 리베이스는 커밋 해시를 변경하므로, 여러 사람이 동시에 공유하는 브랜치에서는 팀 합의/주의 하에 진행한다.

### 4.2 보호 규칙

- `main`, `dev`는 **직접 push 금지**, PR(리뷰) 기반 머지 권장
- `main` 머지는 스프린트 종료 담당자(또는 릴리즈 담당자)가 수행

---

## 5. 요약

- `main`: 배포 기준, 스프린트 종료 후만 머지
- `dev`: 스프린트 중 완료 이슈를 통합하는 개발 브랜치
- `<kan-이슈번호>-이름`: 이슈 단위 개발/테스트 브랜치
- `<개인이니셜>/<브랜치명>`: 개인 작업 브랜치(예: `hs/1234`)
