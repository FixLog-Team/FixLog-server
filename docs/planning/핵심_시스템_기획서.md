## 0. 문서 목적 및 전제

### 0.1 문서 목적

본 문서는 FixLog 1차 MVP에서 제공되는 **개인 문서 보관함 기능**에 대한 시스템 및 기능 동작을 정의한다.

본 문서는 UI 상세 설계가 아닌,

- 데이터 구조
- 기능 책임
- AI 처리 흐름
- 검색 및 인덱싱 전략

을 명확히 하는 것을 목적으로 한다.

### 0.2 전제 조건

- 사용자는 로그인된 상태이다.
- 모든 문서는 **개인 범위(private)** 로 관리된다.
- 1차 MVP에서는 개인 사용을 전제로 하며, 공유 기능은 제공하지 않는다.

---

## 1. 핵심 개념 정의

### 1.1 Document (문서)

사용자가 작성한 하나의 지식 단위.
트러블슈팅, 오류 원인 분석, 해결 과정, 회고 메모 등을 포함한다.

### 1.2 Block

문서를 구성하는 최소 편집 단위.
문서는 노션처럼 여러 개의 Block으로 구성되며, 블록 단위로 저장된다.

### 1.3 Metadata

문서 검색, 정렬, AI 처리 효율을 위한 보조 데이터.

예:

- 사용자 입력 태그
- 자동 추출 키워드
- 생성/수정 시점

### 1.4 AI Artifact

AI 처리 전반에서 생기는 모든 파생 데이터

예:

- 문서 요약 결과
- 문서 임베딩 벡터
- 검색용 문장 매핑 정보 (관련 문서 검색 시, 문서의 **어느 문장이랑 관련 있는지)**

---

## 2. 개인 문서 보관함의 책임 범위

### 2.1 포함 범위 (1차 MVP)

- 개인 문서 저장 및 관리
- 블록 기반 문서 구조 보관 (노션)
- AI 기반 문서 요약 생성 및 적용
- 개인 문서 대상 관련 문서 검색
- 문서 버전 히스토리 관리
- 문서 인덱싱

### 2.2 제외 범위 (1차 MVP)

- 문서 공유 / 팀 문서
- 실시간 협업
- 퍼블릭 문서 공개
- 자동 태그 고도화

---

## 3. 사용자 관점 기능 흐름 (High-level)

```
1. 사용자가 문서를 작성함 (메모리/클라이언트 상태)
  1-1. 여태까지 작성된 내용을 기반으로 AI 문서 요약 가능
2. 사용자가 저장 버튼을 누름
3. 문서 데이터가 DB에 저장됨 (블록 JSON과 각 블록의 Text 저장)
	3-1. AI 분석이 비동기로 시작됨 (2차 MVP 고려)
4. 사용자가 문서를 탐색/검색함
```

## 4. 기능별 상세 정의

---

## 4.1 문서 목록 조회 (개인 문서 보관함)

### 목적

사용자가 본인이 작성한 문서들을 한눈에 탐색할 수 있도록 한다.

### 기능 정의

- 로그인한 사용자의 문서 목록을 조회한다.
- 문서는 리스트 형태로 나열된다. (Gmail 참고)
- 각 문서는 최소한의 메타데이터를 포함한다.

### 포함 데이터 예

- document_id
- title
- summary (존재할 경우)
- last_updated_at

---

## 4.2 문서 조회 및 블록 기반 편집

### 목적

트러블슈팅 과정을 자유롭게 기록할 수 있도록 한다.

### 기능 정의

- 문서는 블록 기반 구조로 저장된다.
- 사용자는 텍스트를 자유롭게 추가/수정할 수 있다.
- 문서는 정리되지 않은 상태로도 저장 가능하다.

### 설계 원칙

- 초기 입력은 난잡해도 허용한다.
- 문서의 구조화는 AI 요약 단계에서 수행한다.

---

## 4.3 AI 기반 문서 요약 기능

### 목적

- 난잡한 메모를 의미 단위로 정리한다.
- 문서 재사용성과 검색 품질을 향상시킨다.

### 트리거

- 사용자가 “문서 요약” 버튼을 실행한다.

### 처리 흐름

1. 원본 문서(Block JSON)를 기준으로 AI 요약 요청
2. AI는 정리된 문서 초안을 생성
   1. 이때, Block Json을 기반으로 문서를 생성해줘야 한다

### 결과 처리

- 좌측 영역: 기존 문서
- 우측 영역: AI가 생성한 정리 문서
- 사용자는 다음 중 하나를 선택한다.
  - 수락: AI 문서를 새로운 본문으로 저장
    - 이전 문서는 History로 남게 되고, 해당 문서의 과거 내역 페이지에서 이전 문서를 볼 수 있다.
    - 과거 내역은 최신 순으로 나열되고, 스크롤을 내릴 수록 과거 내역이 나온다.
    - 각 내역마다 **롤백** 버튼이 존재하고, 클릭 시 해당 내용으로 복원할 수 있다.
  - 거절: 기존 문서를 유지

### 설계 원칙

- AI 결과는 항상 “제안”이다.
- 원본 문서는 항상 보호된다.

---

## 4.4 AI 요약 방식에 대한 MVP 판단

### 기본 원칙

- AI는 ‘문서 요약’ 트리거 시점에 작성된 문서 전체를 기반으로 요약을 수행한다.

### 확장 가능성

- 향후 고도화 단계에서:
  - 추가 설명 입력 영역을 옵션으로 확장 가능
  - 요약 프롬프트 커스터마이징 가능

---

## 4.5 AI 기반 개인 문서 검색

### 목적

키워드 기반이 아닌 의미 기반 검색을 제공한다.

### 검색 범위

- 로그인한 사용자 본인의 문서만 검색 대상이다.

### 처리 전략

### 1단계: 문서 인덱싱

- 문서 저장 또는 수정 시 임베딩 생성
- Sentence Transformer (snunlp 계열) 사용
- 임베딩 벡터를 벡터 DB에 저장
  > 문서의 의미를 숫자 좌표로 바꾼 것
  **임베딩**은 문장/문서를 “의미가 비슷하면 가까워지는 숫자 묶음”으로 바꾸는 작업이다.

### 2단계: 태그 기반 1차 필터링

- 문서에서 키워드/태그 추출
- Postgres JSONB + GIN Index 활용 (추천)
- 탐색 문서 수를 제한 (토큰 절약 차원)

### 3단계: 의미 기반 검색

- 사용자 검색 문장을 임베딩
- 벡터 유사도 기반 검색 수행
- 상위 N개 문서 반환

### 4단계: 결과 표시

- 가장 연관성 높은 문서 목록 제공
- 관련 문장 또는 요약 정보 표시

---

## 5. 데이터 설계 관점 원칙

### 5.1 데이터 분리 원칙

- 원문(Document / Block)
- 메타데이터
- AI Artifact (요약, 임베딩)

각 데이터는 분리 저장한다.

### 5.2 재생성 가능성 전제

- AI 요약 및 임베딩은 언제든 재생성 가능하다.
- 기존 데이터는 항상 보존한다.

---

## 6. 시스템 처리 흐름 (Sequence)

```
문서 저장
→ Document / Block 저장
→ Metadata 업데이트
→ AI 요약 요청 (비동기)
→ 요약 결과 저장
→ 임베딩 생성
→ 검색 인덱스 업데이트
```

## 7. 보안 및 저장 관련 고려사항

- 문서 내용은 Base64 인코딩 및 압축 후 저장 가능
- SQL Injection 방어 목적
- AI 처리 및 검색 시 디코딩 후 사용

---

## 8. 1차 MVP에서 하지 않는 것

- 문서 공유
- 문서 권한 관리
- 문서 버전 비교
- 자동 태그 고도화
- 실시간 협업
